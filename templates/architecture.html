How your question becomes a query

The pipeline routes your natural-language question to the right data product, assembles relevant context from three vector collections and an in-memory catalog, generates a validated SQL or GraphQL query, and returns it — all without touching actual data.

STEP 1 Product Routing
Your question is converted to a 384-dimension vector using sentence-transformers (all-MiniLM-L6-v2). This vector is compared against all data-product embeddings via cosine similarity in ChromaDB. The top 3 candidates are then reranked by the LLM to select the single best match.
Input
"Show me equity trades on NYSE today"
Output
trade_ledger (confidence: 0.92)
Pipeline
question → embed(384d) → cosine search → top 3 products → LLM rerank → best match
product_id passed downstream
STEP 2 Relationship Relevance Filtering
The selected product's one-hop relationships are scored against your question using embedding similarity + keyword matching. Only relationships above a relevance threshold get full context injected (schema, rules, value dictionaries). Others are mentioned but not expanded — preventing noise in the LLM prompt.
All relationships (trade_ledger)
instrument_master via instrument_id
account_master via account_id
party_master via counterparty_id
party_master via broker_id
After relevance filtering
instrument_master — full context
  (embedding: 0.41, keywords: equity, NYSE)
account_master — mention only
party_master — mention only
relevant schemas + rules selected
STEP 3 Context Assembly
Five pieces are assembled into a single LLM prompt: the primary product's DDL and business rules, relevant related products' full context, few-shot query examples retrieved via vector search, and a brief mention of other available JOINs.
From in-memory catalog
Primary DDL — CREATE TABLE trade_ledger (…)
Primary rules — Confirmed: NOT IN (CXL,REJ)
Related DDL — CREATE TABLE instrument_master (…)
Related rules — Active: status = 'A'
Value lookups — asset_class_code: EQ, FI, FX…
From vector DB (examples collection)
Few-shot example 1:
Q: "Show all trades executed today"
A: SELECT … FROM trade_ledger WHERE …

Few-shot example 2:
Q: "Total buy volume by instrument"
A: SELECT … GROUP BY instrument_id
assembled prompt sent to LLM
STEP 4 LLM Query Generation
The LLM (Mistral 7B locally via Ollama, or Claude Sonnet on AWS Bedrock for production) receives the assembled prompt and generates a Snowflake SQL or GraphQL query. It applies business rules, uses correct physical column names from the provided schemas, and follows patterns from the few-shot examples.
Generated SQL
SELECT t.trade_id, im.instrument_name, t.quantity, t.price
FROM trade_ledger t
JOIN instrument_master im ON t.instrument_id = im.instrument_id
WHERE im.asset_class_code = 'EQ'
  AND im.exchange_code = 'NYSE'
  AND t.trade_date = CURRENT_DATE()
  AND t.status_code NOT IN ('CXL', 'REJ')
ORDER BY t.booking_date DESC;
generated query passed to validator
STEP 5 Validation & Self-Correction
The generated query is parsed by sqlglot (Snowflake dialect). Every column reference is checked against the schemas of all included products — both the explicit attributes list and columns extracted directly from DDL. Dangerous operations (DROP, DELETE, etc.) are blocked. If validation fails, the error is fed back to the LLM for up to 2 retries.
Checks performed
1. SQL syntax parse (sqlglot / Snowflake)
2. Column existence vs attribute list
3. DDL-aware column extraction fallback
4. Cross-product column validation
5. Guardrails (block DROP, DELETE, INSERT, UPDATE)
Self-correction loop
attempt 1 → Column 'name' not found
error fed back → LLM retries with feedback
attempt 2 → Valid query returned
max retries: 2

Where does the context come from?

Collection 1
Products
Product-level embeddings for routing. One document per data product with name, description, and key attributes.
ChromaDB
Collection 2
Attributes
Attribute-level embeddings for fine-grained schema retrieval. Used for products with many columns.
ChromaDB
Collection 3
Query Examples
Natural-language questions embedded for few-shot retrieval. SQL/GraphQL stored in metadata.
ChromaDB
In-Memory
Schema DDL / SDL
Full CREATE TABLE or GraphQL SDL per product. Injected directly into the LLM prompt at generation time.
Catalog
In-Memory
Business Rules
